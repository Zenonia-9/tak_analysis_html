Executive summary

The module is an application-level business customization for Akaree/Tunn operations. It is not limited to one Odoo app: it links contacts, sales, inventory, delivery routing, stock requisitions, consignment, POS closing inventory, events/promotions, accounting reports, payroll compensation, planning, project tasks, and sale commissions.

The heaviest custom areas are Sales, Inventory/Batch Picking, Route Delivery/Mobile API, and Reporting. The addon mostly extends original Odoo models by inheritance, while also defining custom operational models such as route.detail, tak.stock.check, tak.partner.contract, tak.mobile.user, and several delivery/requisition line models.

Manifest dependencies: base, contacts, stock, account, stock_picking_batch, base_geolocalize, mail, stock_account, hr, hr_payroll, hr_contract, sale, sale_stock, pos_sale, planning, project, hr_timesheet, sale_project, sale_commission.

Business workflows implemented

No.WorkflowWhat the module does
1Master data and address hierarchyCustomers/vendors receive generated IDs; contacts are classified by business partner type/category and enriched with district, township, ward, and map URL fields.
2Partner contractsContracts assign stage by date, calculate a reminder date three months before expiry, and scheduled logic sends an email when the reminder date arrives.
3Sales order controlSales orders validate that storable consumable products belong to the chosen brand, validate commission presence, compute commission deductions, propagate remarks to stock moves and invoices, and apply analytic distribution by branch/product category.
4ConsignmentSales can be marked as consigned sale or consigned internal transfer, choose source/destination locations, enforce consignment location value limits, and create refill sale orders from posted invoices.
5Batch requisitionBatch transfers are office-approved, assigned to driver/vehicle/schedule, then generate grouped requisition product lines, invoice links, sale-order links, and printable totals.
6Route and mobile deliveryRoutes are created from pickings or customers, moved through draft/ready/on-going/done, converted into project map tasks, and populated with delivery status, product totals, and invoice totals from mobile app updates.
7Stock checks and pre-sale visitsMobile endpoints create stock check records, counted lines, pre-sale visit action lines, and new customer entries tied to route/customer context.
8POS and event closingPOS sessions can generate closing inventory snapshots; events can aggregate POS order lines and cash statement lines for event/promotion reporting.
9Payroll compensationEmployee compensation records and salary-rule flags let employee forms show totals and drill-down actions for commission, incentive, overtime, fine, other income, and deduction.
10ReportingCustom QWeb reports cover invoice, official receipt, event, sale order, stock requisition, purchase order, delivery order, and delivery slip; analysis models gain branch, brand, partner type/category, customer ID, payment method, and commission fields.

Technical observations

  • The manifest is still versioned as 18.0.1.0.1 while the project is Odoo 19; the module has already had some Odoo 19 compatibility work, but the version string still communicates Odoo 18 origin.
  • controllers/controllers.py imports odoo.addons.restful but restful is not declared in depends. If restful is not installed before this addon, controller import will fail at module load time.
  • Several XML files exist but are not loaded by the manifest: data/cron.xml, follow-up templates, SWM legacy templates, report_bak, static assets, and the commented vehicle view. They are documented as retained/legacy material rather than active runtime behavior.
  • The manifest assets section comments out SCSS and JS assets, so static/src/js/map_renderer.js and TAK/stock SCSS files are present but inactive unless re-enabled.
  • Some inherited views reference optional modules such as website_event, sale_planning, and project_forecast while those are not explicit manifest dependencies. They may be satisfied by installed enterprise modules, but they are still integration points to verify on install/update.
  • This report is documentation only and is not referenced by __manifest__.py, so it does not change loaded Odoo behavior.
